### Locked Decisions

**Orchestration & Run Model**
- D-01: Self-hosted n8n on the same VPS as the Python service. n8n handles scheduling/triggers, Gmail polling, Claude API calls (HTTP Request node), email sending (SMTP node), and Telegram notifications. No APScheduler needed.
- D-02: Python FastAPI service exposes Phase 1 logic as HTTP endpoints. n8n calls these endpoints as part of the workflow. Python also owns all capabilities n8n can't run natively: JobSpy scraping, Kalibrr scraping, SQLite DB reads/writes, Phase 3 Camoufox browser automation.
- D-03: JobSpy scraping (INGEST-03) is a Python FastAPI endpoint (`POST /scrape-jobspy`) triggered by an n8n cron schedule. Same pattern for Kalibrr scraping (`POST /scrape-kalibrr`).

**Gmail Ingestion (INGEST-01)**
- D-04: Gmail is polled via the Gmail API — not Pub/Sub push. Polling interval: every 1 hour. No GCP project, no Pub/Sub topic, no public webhook endpoint required.
- D-05: Job alert emails are identified by sender filter: `jobalerts-noreply@linkedin.com`. No Gmail label or subject-line matching.
- D-06: Processed messages are tracked via the Gmail API's `historyId`. After each poll, store the latest `historyId` in the DB. Next poll fetches only messages since that ID — zero API calls when no new mail.
- D-07: Gmail OAuth uses an offline refresh token stored in `.env` (never committed). Run the OAuth flow once locally to obtain the refresh token; the agent silently refreshes access tokens at runtime.
- D-08: LinkedIn job alert emails are digest-style (one email, multiple job listings). The parser must handle 1-to-many jobs per email — treat every email as potentially containing multiple listings.

**Email Parsing (INGEST-01)**
- D-09: AI-assisted extraction — n8n strips the email to plain text, then calls Claude Haiku 3.5 via HTTP Request node with a structured output prompt. Claude returns a JSON array of job listings.
- D-10: Extracted fields per job: `title`, `company`, `location`, `url`. No JD text from the email — JD is stored from subsequent enrichment or left blank.
- D-11: LinkedIn tracking URLs (e.g. `linkedin.com/comm/jobs/view/...`) are stored as-is. The Phase 1 dedup already strips tracking params during canonicalization. No extra HTTP redirect-follow at parse time.

**AI Model Selection**
- D-12: Claude Haiku 3.5 for all extraction and selection tasks: email parsing, resume selection, screening question answers.
- D-13: Claude Sonnet 4.x for cover letter generation only. ~$0.015/application. Estimated total cost: ~$20/1,000 applications.
- D-14: Both models are called by n8n via HTTP Request nodes using the Anthropic API. Claude API key stored in n8n credentials.

**Resume Library & AI Selection (AI-01)**
- D-15: Resume files are a mix of `.docx` and `.pdf`. Resumes are differentiated by role type / industry focus.
- D-16: Resume files are committed to the private git repo and deployed with the code. They live at `resumes/`.
- D-17: Resume selection uses LLM prompt comparison via Haiku: send the job description + a summary of each resume's focus area, ask which best fits. No embeddings infrastructure needed.
- D-18: A Python FastAPI endpoint (`POST /select-resume`) handles selection and returns both the chosen resume name and its text content.

- D-19: n8n builds the cover letter prompt from: job description (from parsing step) + resume text (from `/select-resume`) + Stefano's profile. n8n calls Claude Sonnet API and receives the cover letter text.

## Summary

Phase 2 wires five new capability groups into the Phase 1 foundation: Gmail ingestion, scraper-based lead collection, AI generation, email application submission, and Telegram notifications. The architecture splits cleanly between **n8n** (orchestration, scheduling, external API calls, notifications) and **Python FastAPI** (all capabilities requiring Python libraries: JobSpy, Kalibrr, SQLite writes, Phase 1 filter/dedup, resume text extraction).

The critical architectural resolution for this phase is that **CLAUDE.md recommends LangGraph; CONTEXT.md decisions lock n8n**. The research conclusion is: n8n is the correct choice for Phase 2's specific workload. Phase 2 is primarily orchestration of HTTP calls, scheduled scraping, and notification delivery — exactly the use case n8n excels at. LangGraph would be superior for Phase 3 (stateful browser automation with complex conditional branching per job type). There is no technical incompatibility: Phase 1 built pure Python logic behind a DB layer that FastAPI will expose. n8n calls FastAPI; both run in the same Docker Compose network.

The second key finding is that n8n's built-in Gmail Trigger node (polling-based, ~1-minute intervals) is a viable simpler alternative to a custom Python Gmail API + historyId implementation. The decision (D-04, D-06) to implement historyId tracking in Python is locked and technically sound — it gives explicit control and zero risk of n8n's documented duplicate-processing edge cases. The FastAPI `/poll-gmail` endpoint approach is the right call.

**Primary recommendation:** Deploy n8n 2.x + FastAPI in Docker Compose on the same network. n8n owns all scheduling, external API calls, and notifications. FastAPI exposes five endpoints that bridge n8n to Phase 1 Python logic. Use n8n's built-in Anthropic node (or HTTP Request node) for Claude API calls with structured JSON output via the `output_config.format` parameter.

### Pattern 3: Gmail historyId Polling (Python)

**What:** The Gmail poller runs when n8n calls `POST /poll-gmail`. It fetches changes since the stored `historyId`, filters for `messageAdded` events from the target sender, and returns new message IDs. The `historyId` is stored in a new `agent_config` table in SQLite.

```python
# Source: googleapis.github.io/google-api-python-client/docs/dyn/gmail_v1.users.history.html
def poll_gmail_since(service, start_history_id: str, sender_filter: str) -> tuple[list[str], str]:
    """Returns (new_message_ids, new_history_id)."""
    changes = []
    page_token = None
    latest_history_id = start_history_id

    while True:
        response = service.users().history().list(
            userId="me",
            startHistoryId=start_history_id,
            historyTypes=["messageAdded"],
            pageToken=page_token,
        ).execute()
        
        latest_history_id = response.get("historyId", latest_history_id)
        for record in response.get("history", []):
            for msg in record.get("messagesAdded", []):
                changes.append(msg["message"]["id"])
        
        page_token = response.get("nextPageToken")
        if not page_token:
            break

    # Fetch each message to filter by sender — historyTypes doesn't support sender filter directly
    matching_ids = []
    for msg_id in changes:
        msg = service.users().messages().get(
            userId="me", id=msg_id, format="metadata",
            metadataHeaders=["From"]
        ).execute()
        headers = {h["name"]: h["value"] for h in msg["payload"]["headers"]}
        if sender_filter in headers.get("From", ""):
            matching_ids.append(msg_id)

    return matching_ids, latest_history_id
```

**404 handling:** If `history.list()` returns HTTP 404, the historyId has expired. Fall back to `messages.list(q=f"from:{sender_filter}")`, get the most recent message's historyId, and store it as the new checkpoint.

**Confidence:** HIGH [CITED: developers.google.com/workspace/gmail/api/reference/rest/v1/users.history/list]

### Pattern 6: Resume Text Extraction for LLM Comparison

**What:** FastAPI reads all files in `resumes/` and extracts plain text per file. Text is then included in a Haiku prompt asking which resume best matches the job description.

def extract_resume_text(filepath: str) -> str:

### Anti-Patterns to Avoid

- **Calling `localhost` in n8n HTTP Request nodes:** n8n runs in a Docker container. Use the Docker service name (`http://api:8000`), not `localhost`. `localhost` resolves to the n8n container itself.
- **Storing n8n encryption key after first run starts:** `N8N_ENCRYPTION_KEY` must be set before the first launch. Changing it after credentials exist requires re-entering all credentials.
- **Using n8n 1.x SQLite default for credentials in a multi-container setup:** The n8n encryption key encrypts credentials stored in n8n's own SQLite. Use Postgres if n8n needs to scale, but SQLite is fine for single-node MVP.
- **Running python-jobspy scrape_jobs() in an async endpoint without executor:** Blocks the FastAPI event loop and hangs all concurrent requests during the scrape.
- **Storing Gmail historyId only in memory:** Process restart loses the checkpoint and causes full re-poll or missed emails. Store in SQLite `agent_config` table.
- **Using n8n Gmail Trigger node for D-06 historyId tracking:** The n8n Gmail Trigger uses timestamp-based dedup internally (not historyId). It has documented edge cases with duplicate processing on restart. The custom Python + historyId approach (D-04, D-06) avoids this.

## Common Pitfalls

### Pitfall 1: historyId Expiry Without Fallback

**What goes wrong:** The Gmail `historyId` stored in the DB becomes invalid after 7+ days of inactivity (rare cases: a few hours). `history.list()` returns HTTP 404. Without a fallback, the poller crashes silently — no new emails processed.

**Why it happens:** historyIds have an expiry window. A long weekend or network outage can invalidate the stored checkpoint.

**How to avoid:** Always wrap `history.list()` in a try/except for `googleapiclient.errors.HttpError` with status 404. On 404: fall back to `messages.list(q="from:jobalerts-noreply@linkedin.com is:unread")`, get the historyId from the most recent message, store as the new checkpoint, and return empty list for this poll cycle.

**Warning signs:** Log entries showing `HttpError 404` in Gmail API calls.

---

### Pitfall 2: n8n Encryption Key Loss = Credential Wipeout

**What goes wrong:** The `N8N_ENCRYPTION_KEY` is regenerated (or Docker volume deleted), wiping all stored credentials — Anthropic API key, Gmail OAuth, SMTP password, Telegram bot token. All workflows stop immediately.

**Why it happens:** n8n encrypts all credential values using this key. Changing the key means it can no longer decrypt existing credentials.

**How to avoid:** Generate the key once (`openssl rand -hex 32`), store it in `.env` (gitignored but backed up), and treat it like a root CA key. Never regenerate. Back up the `n8n_storage` Docker volume alongside it.

**Warning signs:** Workflow executions failing with "Invalid credential" errors after a re-deploy.

---

### Pitfall 3: python-jobspy Returns Empty on IP Block

**What goes wrong:** `scrape_jobs()` returns an empty DataFrame with no error. Indeed has silently blocked the VPS IP. The system logs 0 new jobs, no alert fires, and Stefano thinks there are simply no new leads.

**Why it happens:** Indeed does not return a 4xx error on detection — it silently returns empty results or a minimal result set.

**How to avoid:** After every `/scrape-jobspy` run, check result count. If count == 0 AND the search was broad (should have returned results), fire a Telegram OPS alert: "JobSpy returned 0 results — possible IP block." Log the zero-result event in the audit trail. Consider residential proxy rotation if this occurs repeatedly.

**Warning signs:** Zero-count scrape results for 2+ consecutive runs on a broad search term.

---

### Pitfall 4: Email Digests Producing 0-Job Parse (Claude Structured Output)

**What goes wrong:** Claude returns `{"jobs": []}` for a LinkedIn digest email that visually contains 5 job listings. This happens when: (a) the email body was not properly stripped to plain text before sending to Claude, (b) the prompt is too restrictive, or (c) Claude received HTML instead of text.

**Why it happens:** The n8n Gmail Trigger returns HTML body by default. If the HTML is sent raw to Claude without stripping, the model may not recognize the job listings embedded in `<a>` tags and surrounding boilerplate.

**How to avoid:** Strip HTML to plain text in n8n before the Claude call (n8n Code node or HTTP call to a `/strip-html` FastAPI utility). Validate parsed job count in n8n: if `$json.jobs.length === 0` and the email body length > 500 chars, fire an OPS alert and log the raw email body for debugging.

**Warning signs:** Email received in Gmail (visible manually), audit log shows no QUEUED entries from that email's timestamp window.

---

### Pitfall 5: FastAPI Startup Fails to Find SQLite DB on First Run

**What goes wrong:** FastAPI starts before the `data/` directory exists in the Docker volume, and `init_db()` fails because the path doesn't exist. The API container crashes on startup.

**Why it happens:** Docker volume mounts create the parent directory, but if `data/` is not created in the Dockerfile or by `init_db()`, the SQLite path fails.

**How to avoid:** Phase 1's `db.py` already handles `parent.mkdir(parents=True, exist_ok=True)`. Verify the Docker volume mount maps to the correct path the API reads from `.env`. The `DB_PATH` env var must point to the volume-mounted path (`/data/jobs.db`), not a local path.

**Warning signs:** Container exits immediately with `FileNotFoundError` or SQLite `unable to open database file`.

---

### Pitfall 6: n8n HTTP Request Node Uses `localhost` Instead of Service Name

**What goes wrong:** n8n workflows configured with `http://localhost:8000/ingest-lead` fail with "Connection refused" because n8n resolves `localhost` to its own container.

**Why it happens:** Docker containers are network-isolated. `localhost` in an n8n container is the n8n container, not the FastAPI container.

**How to avoid:** All n8n HTTP Request nodes must use `http://api:8000` (the Docker Compose service name). Document this prominently in the n8n workflow import instructions. If testing locally outside Docker, use `http://host.docker.internal:8000` (macOS/Windows) or the host's network IP.

**Warning signs:** n8n execution log shows "ECONNREFUSED" or "Network request failed" immediately on HTTP Request nodes.

---

### Pitfall 7: Email Send Fails Due to Gmail "Less Secure App" Block

**What goes wrong:** n8n Send Email node with SMTP + Gmail fails with auth error. Google blocked basic SMTP auth in 2023.

**Why it happens:** Google deprecated password-based SMTP in May 2023. Standard SMTP credentials no longer work with `smtp.gmail.com`.

**How to avoid:** Use n8n's **Gmail node** (OAuth2-based Send Email) instead of the generic SMTP Send Email node. The Gmail node uses OAuth2 credentials stored in n8n, bypassing the basic auth restriction. Alternative: use an SMTP relay service (Mailgun, SendGrid free tier) for a non-Gmail sending path.

**Warning signs:** Send Email node fails with "Username and Password not accepted" from smtp.gmail.com.

### FastAPI `/ingest-lead` Endpoint (uses Phase 1 logic)

```python
# Source: Phase 1 main.py patterns + FastAPI dependency injection [ASSUMED]
from fastapi import APIRouter, Depends
from sqlalchemy.ext.asyncio import AsyncSession
from src.filter.eligibility import check_eligibility
from src.filter.dedup import is_duplicate, hash_url
from src.audit_log import write_audit, AuditEvent
from src.queue.models import Job, JobStatus

router = APIRouter()

class LeadIn(BaseModel):
    url: str
    title: str
    company: str
    location: str | None = None
    source: str
    clean_jd: str | None = None
    apply_type: str | None = None

@router.post("/ingest-lead")
async def ingest_lead(payload: LeadIn, session: AsyncSession = Depends(get_session)):
    url_hash = hash_url(payload.url)
    async with session.begin():
        if await is_duplicate(session, payload.company, payload.title, payload.location, url_hash):
            await write_audit(session, source=payload.source, event=AuditEvent.DEDUP_SKIP)
            return {"status": "duplicate", "job_id": None}
        
        result = check_eligibility(payload.title, payload.location, payload.clean_jd, config)
        status = JobStatus.QUEUED if result.passed else JobStatus.REJECTED
        job = Job(url=payload.url, url_hash=url_hash, ...)
        session.add(job)
        await write_audit(session, source=payload.source, event=AuditEvent.QUEUED, job_id=job.id)
    
    return {"status": status.value, "job_id": job.id}
```

## Architecture Conflict Resolution: n8n vs LangGraph

**The conflict:** CLAUDE.md (Phase 1 research) recommends LangGraph. CONTEXT.md decisions (D-01, D-02) lock n8n. This is not a mistake — the decisions were made with awareness of CLAUDE.md.

**Why n8n is correct for Phase 2:**

Phase 2 work is: scheduled HTTP calls, external API calls (Gmail, Claude, JobSpy), email sending, and Telegram notifications. This is exactly the workflow-orchestration-of-HTTP-calls use case n8n was designed for. n8n's HTTP Request node, Schedule Trigger, Send Email node, and Telegram node eliminate hundreds of lines of Python that would otherwise be written.

**Why LangGraph remains the right call for Phase 3:**

Phase 3 adds stateful browser sessions, conditional branching per form type, crash recovery, and human-in-the-loop for CAPTCHA challenges. This is the stateful agentic decision-making use case LangGraph excels at. The hybrid architecture (n8n orchestrates; Python FastAPI provides the stateful logic) cleanly separates these concerns.

**Technical compatibility:** n8n calling FastAPI endpoints is a standard HTTP integration. There is zero technical incompatibility between n8n and the Phase 1 SQLAlchemy + SQLite codebase.

**Recommendation:** No change to locked decisions. Proceed with n8n + FastAPI as specified in CONTEXT.md.
